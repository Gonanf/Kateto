# Backlog

Read and update the canonical product backlog through the authorized backlog events. Preserve existing fields, use explicit priorities and statuses, and confirm the resulting item or filter.

## MoSCoW prioritization

Classify every backlog item into one of four tiers:
- **Must have**: non-negotiable for delivery. Without it the project fails.
- **Should have**: important but a workaround exists if dropped.
- **Could have**: nice-to-have, included only if time and budget allow.
- **Won't have (this time)**: explicitly deferred. Record the reason.

Prioritization is collaborative — involve stakeholders and the delivery team. Revisit priorities each grooming session.

## DEEP backlog principle

A healthy backlog is:
- **Detailed appropriately**: near-term items are well-specified; far-term items are coarse
- **Estimated**: every item has a rough size (story points or t-shirt)
- **Emergent**: items can be added, removed, or re-prioritized as understanding grows
- **Prioritized**: the order reflects business value and dependency constraints

## Task decomposition

Break large items (epics) into smaller, independently deliverable tasks. Each task should:
- Have a clear definition of done
- Be estimable (hours or story points)
- Be actionable by one person or pair
- Deliver value independently if possible

## Refinement cadence

Schedule regular backlog grooming (every 2-3 sprints or weekly). Each session:
1. Review the top of the backlog
2. Re-prioritize based on new information
3. Split oversized items
4. Re-estimate if understanding changed
5. Remove items that no longer add value

## Estimation guidelines

- Use relative sizing (story points), not hours
- A point is effort, not time — account for complexity, risk, and uncertainty
- If an item is too large to estimate (over 13 points), split it
- Re-estimate when new information changes the scope
- Keep estimation separate from priority: a 1-point Must and a 13-point Must are both Musts
